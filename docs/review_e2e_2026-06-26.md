## 2. 发现的问题

### 2.1 P0: 工具注册断层

**现象**: Generator 的 prompt 告诉 LLM 有 6 个工具可用，但运行时 registry 中只有 3 个。

**根因**: `code_ops.py`（注册 `search_code`）和 `test_ops.py`（注册 `run_tests`、`run_lint`）没有被 Generator 的依赖链导入。这两个模块只在 `evaluator.py` 被导入时才触发注册，但此时 Generator 已经运行完毕。

```
导入链:
  graph.py → GeneratorAgent → file_ops (注册 read_file, list_directory, write_file)
  graph.py → EvaluatorAgent → test_ops  (注册 run_tests, run_lint)
                            → code_ops  (注册 search_code)  ← Generator 运行时还没到这步
```

**影响**: 如果 LLM 在生成阶段调用 `run_tests`、`run_lint` 或 `search_code`，`registry.execute()` 抛出 `ValueError("Tool 'xxx' not found")`，被 `safe_execute` 静默吞掉，Generator 产出空结果。

**修复**: `src/workflow/graph.py` 顶部显式导入所有工具模块：

```python
import src.tools.code_ops  # noqa: F401
import src.tools.file_ops  # noqa: F401
import src.tools.test_ops  # noqa: F401
```

### 2.2 P0: Token 预算计费错误

**现象**: Generator 在第 10 次迭代时因 token 预算耗尽而停止，只完成了 model + schema 的写入，service / router / tests 均未实现。

**根因**: `LoopController.check_token_budget()` 检查的是 `LLMClient._total_tokens`（全局累加值），而非本轮 Generator 的实际消耗。Planner 调用 DeepSeek 消耗的 ~2.3k tokens 被计入了 Generator 的预算。

```
修复前:
  Planner:  0 → 2,361 tokens
  Generator: 2,361 → 111,382 tokens  ← 预算 100k，但基线已被 Planner 抬高
  实际可用: 100,000 - 2,361 = 97,639 tokens

修复后:
  Generator: baseline 重置为 2,361，delta 计算从 0 开始
  实际可用: 完整 150,000 tokens
```

**修复**:

1. `src/harness/loop/controller.py` — 新增 `_token_baseline` 字段和 `reset_token_baseline()` 方法，`check_token_budget()` 改为检查 delta：

```python
def check_token_budget(self, used: int) -> bool:
    self._total_tokens = used
    return (used - self._token_baseline) < self.token_budget
```

2. `src/agents/generator.py` — 循环入口调用 `reset_token_baseline()`：

```python
self.loop_controller.reset_token_baseline(self.llm.total_tokens_used)
```

3. `src/config.py` + `.env` — `TOKEN_BUDGET` 从 100,000 提升到 150,000（完整 CRUD 任务需要 ~99k）。

### 2.3 观察: Generator 非确定性行为

在调试过程中观察到 Generator 的 LLM 响应具有非确定性：

- 同样的 prompt + 同样的 codebase，不同运行产生不同数量的工具调用（17 ~ 37 次）
- 有时 LLM 直接返回文本而不调用工具（"放弃"行为），导致 `execution_trace` 为空
- 重试次数用尽后，LLM 倾向于直接返回文本而非继续尝试

这不是 bug，是 LLM 的固有特性。测试设计已考虑此点，接受 escalation 为合法结果。

## 3. 新增测试

### 3.1 文件结构

```
tests/e2e/
├── __init__.py
└── test_real_deepseek.py      # 3 个测试，全部调真实 DeepSeek API
```

### 3.2 测试用例

| 测试 | 作用 | 超时 |
|------|------|------|
| `test_planner_returns_valid_json` | 烟雾测试：只跑 Planner，验证返回可解析的 plan + contract | 60s |
| `test_full_pge_loop` | 核心测试：完整 PGE 循环，验证 PASS 或 escalation | 300s |
| `test_pge_loop_produces_meaningful_trace` | 可观测性：验证执行记录包含 read_file / write_file 调用 | 300s |

### 3.3 设计原则

- **两种结果都算通过**: PASS（所有 criterion 满足）和 ESCALATE（重试耗尽）都是系统正常工作的证明。测试验证的是"系统跑完了没有崩溃"，不是"LLM 一定能写出完美代码"。
- **隔离**: 每次运行复制 `toy-repo/` 到临时目录，不污染原始 codebase。
- **独立图实例**: 每个测试调用 `build_graph().compile()` 创建新实例，不共享单例。

### 3.4 运行方式

```bash
# 只跑 E2E（需要真实 API key）
pytest tests/e2e/ -v -m integration --timeout=300

# 跳过 E2E，只跑单元/集成测试
pytest tests/ -v -m "not integration"
```
